This page is intentionally a grab bag. It exists to show what the article system can already render without forcing you to rediscover the syntax every time you write a new post.

You can use standard markdown for almost everything:

- headings
- paragraphs
- lists
- blockquotes
- fenced code blocks
- markdown tables
- links
- images

You can also mix in a small set of custom MDX components when plain markdown stops being expressive enough.

<Callout title="Practical rule">
  Start with plain markdown. If a section begins to feel awkward, repetitive, or visually unclear, then upgrade just that section to a richer MDX block.
</Callout>

## Plain markdown still does most of the work

Regular markdown paragraphs are still the main path. That is deliberate. It keeps writing portable and lowers the cognitive overhead of drafting.

## Suggested workflow for writing

1. Create a new folder under `content/blog/<slug>/`.
2. Add `index.mdx` with frontmatter and your draft.
3. If a section needs structured data, add `data.ts` next to the MDX file.
4. Import only the data that article needs.
5. Run the local build before pushing.

## When to use MDX instead of plain markdown

Use MDX when:

- you need a reusable custom block
- you need richer visual structure than markdown alone can express
- a small interactive widget materially improves the explanation

Do not use MDX just because you can. For ordinary prose, markdown is the cleaner format.
