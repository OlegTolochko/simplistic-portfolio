<Callout title="Context">
  This project started as a reimplementation of the paper <em>Training a convolutional neural network to conserve mass in data assimilation</em>, but it evolved into a broader research pipeline: physically constrained simulation, ensemble-based assimilation, learned corrections, and uncertainty quantification on top.
</Callout>

## Overview

The core problem is deceptively narrow: improve data assimilation in a shallow water model while respecting the physical structure of the system. In practice, that becomes a multi-stage research problem involving simulation, observation generation, classical assimilation baselines, learned correction models, and calibrated uncertainty estimates.

This repository ties those pieces together into one end-to-end workflow. The project uses the **MSW** shallow water model as the physical backbone, compares **EnKF** and **QPEns** style assimilation, trains a **CNN** to learn mass-conserving corrections, and then evaluates **CP** and **CQR** based uncertainty quantification on top of the learned system.

## Why the project is interesting

What makes the project compelling is that it sits at the boundary between numerical modeling and machine learning.

- The simulation itself is not an arbitrary toy problem. It has physical structure that should not be ignored.
- The learned model is not asked to replace everything. It is asked to improve a specific part of the pipeline.
- The uncertainty layer is not cosmetic. It matters because the system is ultimately making structured corrections to a dynamical process, and those corrections need calibrated confidence estimates.

That combination makes the repository a strong example of scientific ML done as a pipeline rather than as an isolated model notebook.

## 1. Physical model backbone

The whole system begins with the **MSW model**, which provides the controlled physical environment. The state tracks variables such as velocity, water height, and rain over a spatial grid and evolves them under shallow-water dynamics with stochastic perturbations.

That matters because the later ML components are only meaningful if the upstream dynamics remain physically grounded. The model is not just producing arbitrary tensors; it is defining the space in which the data assimilation and correction tasks make sense.

## 2. Training-data generation for data assimilation

Once the physical simulator is in place, the next step is to generate training data that makes the correction problem learnable. The repository simulates truth trajectories, constructs noisy and masked observations, and runs multiple assimilation variants so their outputs can be compared later.

This stage is especially important because it defines the supervision signal. The CNN is not trained from raw truth alone; it is trained to map the more accessible **EnKF** behavior toward the stronger **QPEns** target.

## 3. CNN and quantile-regression training

The learned part of the pipeline is intentionally constrained. Rather than replacing the physics or the assimilation stack, the network learns a correction from EnKF-style states toward the QPEns target. That is a good decomposition because it turns the learning problem into a structured residual-style refinement rather than a full-system imitation problem.

The repository also includes a quantile-regression variant, which makes it possible to learn interval predictions that later plug naturally into the uncertainty-quantification stage.

<Figure caption="CNN and quantile-regression training pipeline. The important design choice here is that the network learns a correction target, not an unconstrained end-to-end surrogate of the full simulator.">

## 4. Inference loop

At inference time, the system returns to the original dynamical setting: propagate the ensemble, generate observations, assimilate with EnKF, and then apply the learned correction model. Optionally, QPEns can still be computed as a reference for comparison.

That makes the project more than an offline training exercise. The final artifact is a loop that can be run over many inference steps and evaluated operationally against truth and baseline methods.

## 5. History generation for evaluation

To evaluate uncertainty methods properly, the repository generates multi-seed histories across truth, EnKF, QPEns, and CNN-based predictions. This stage is what turns the project from “one trained model” into a real evaluation framework.

It also creates the bridge from raw model behavior to calibrated statistical comparisons. Without this step, the uncertainty work would stay anecdotal.

## 6. Uncertainty quantification

The uncertainty layer is where the project becomes especially interesting from a research perspective.

Two main routes are implemented:

- **Conformal Prediction (CP)**, where non-conformity scores are calibrated and then converted into intervals on held-out data.
- **Conformalized Quantile Regression (CQR)**, where learned lower and upper quantiles are widened just enough to achieve empirical coverage.

The CLI also exposes a broader comparison surface through the `compare_uq_methods` command, including hooks for optional baselines such as MCDO, NIG-based evidential regression, normalized CP, and random-forest-style variants.

## CLI and reproducibility

The repository is organized around a clean CLI surface in `msw_da_ml.main`, which makes the full workflow easier to reason about and rerun. The main steps are:

```bash
uv sync

uv run python -m msw_da_ml.main generate-training-data
uv run python -m msw_da_ml.main train-cnn-model <training_data_name>
uv run python -m msw_da_ml.main train-quantile-regression-model <generated_msw_data_name>
uv run python -m msw_da_ml.main run-inference
uv run python -m msw_da_ml.main generate-conformal-prediction-data
uv run python -m msw_da_ml.main generate-cqr-data
uv run python -m msw_da_ml.main compare-uq-methods <cp_hist_name> <cqr_hist_name>
```

That organization matters. It means the repository reads like a pipeline you can execute, inspect, and extend, rather than a loose collection of research scripts.

## What I would emphasize when presenting this project

If I were presenting this project in conversation or on a CV, I would highlight four things:

1. It combines **physical simulation**, **classical data assimilation**, **neural correction models**, and **uncertainty quantification** in one coherent workflow.
2. The ML component is used in a disciplined way: not as a magical replacement, but as a targeted improvement over a baseline assimilation method.
3. The repo is structured around explicit intermediate artifacts and pipeline stages, which makes it easier to debug and extend.
4. The D2 visualizations are not decorative. They are part of how the project stays understandable despite spanning multiple research ideas.
